# Сохранение пользователя в БД при логине

Дата: 2026-06-04

## Проблема

Информация о пользователях приложения нигде фактически не сохраняется. Таблица
`Users` в БД уже существует (entity `Pet.Jira.Domain.Entities.Users.User`,
зарегистрирована в `ApplicationDbContext`, миграция применена, на неё ссылается
`UserNotification.UserId`), но **в неё никто не пишет** — во всём коде нет ни
одного `_dbContext.Users.Add(...)`. Из-за этого невозможно привязывать к
пользователю настройки на стороне сервера.

Аутентификация выполняется через Jira (basic — username/password, bearer — PAT).
Локальных аккаунтов нет; личность берётся из cookie-claims после входа
(`AuthenticationMiddleware` → `SignInAsync`). Текущие настройки (`UserTheme`,
`UserProfile`) хранятся в браузерном LocalStorage, не на сервере.

## Цель

Завести/обновлять запись пользователя в БД, чтобы позже можно было привязывать к
ней настройки. **Объём этой задачи — только персистентность пользователя.**
Перенос настроек из LocalStorage в БД — отдельная задача в будущем (вне рамок).

## Решение (обзор)

Ленивый идемпотентный upsert пользователя в точке, где приложение разбирает
текущего аутентифицированного пользователя. Реализуется отдельным middleware,
работающим после аутентификации. Это покрывает одной точкой и новые входы, и
уже существующие cookie (см. «Уже авторизованные пользователи»).

## Компоненты

### 1. Доменная модель

`User` (`src/Pet.Jira.Domain/Entities/Users/User.cs`) оставляем без изменений:
`Username` + `Id` / `CreatedAt` / `UpdatedAt` из `BaseEntity`.

Добавляем уникальный индекс на `Username` через
`IEntityTypeConfiguration<User>` в Infrastructure (подхватывается
`ApplyConfigurationsFromAssembly(...)` в `ApplicationDbContext.OnModelCreating`,
`ApplicationDbContext.cs:82`). Индекс защищает от дублей при гонках и повторных
вставках. Под изменение схемы — новая EF-миграция.

### 2. Репозиторий

- `IUserRepository` — интерфейс в Application-слое
  (`Pet.Jira.Application.Users` или соседний namespace), один метод:
  `Task EnsureUserExistsAsync(string username, CancellationToken cancellationToken)`.
- `UserRepository` — реализация в Infrastructure на `ApplicationDbContext`.
  Логика идемпотентна: `SELECT` по `Username`; если записи нет — `INSERT` с
  заполнением `CreatedAt`. Гонку (две параллельные вставки) ловит уникальный
  индекс → перехватываем `DbUpdateException` и трактуем как «запись уже есть»
  (не ошибка).

### 3. Точка вызова — `UserProvisioningMiddleware`

Новый middleware в Web-слое (`src/Pet.Jira.Web/Authentication/`), регистрируется
в конвейере **после** аутентификации (после того как `context.User` заполнен из
cookie) и относительно существующего `AuthenticationMiddleware`.

Логика: если `context.User?.Identity?.IsAuthenticated == true`, берём
`context.User.Identity.Name` (username) и вызываем
`IUserRepository.EnsureUserExistsAsync(...)`, после чего передаёт управление
дальше по конвейеру.

Почему middleware, а не `IdentityService`:
- `IdentityService.CurrentUser` вызывается синхронно
  (`.GetAwaiter().GetResult()`) и часто — обращение к БД оттуда рискованно
  (sync-over-async, нагрузка).
- Middleware оставляет `IdentityService` с единственной ответственностью
  (чтение claims) и даёт ровно один проход provisioning на HTTP-запрос.

### 4. Кэш-гард

Чтобы не обращаться к БД на каждом запросе — кэш уже обработанных username за
время жизни процесса (`IMemoryCache` или потокобезопасная коллекция, напр.
`ConcurrentDictionary`/`HashSet` с синхронизацией).

- Первое появление username → проверка/вставка в БД + занесение в кэш.
- Последующие запросы того же username → мгновенный no-op (проверка кэша).
- После рестарта процесса кэш пуст → на пользователя приходится одна лишняя
  проверка в БД (не вставка — запись уже существует).

## Уже авторизованные пользователи

Полный backfill невозможен: список залогиненных нигде не хранится, личность
живёт только в cookie в браузере. Поймать их можно лишь в момент запроса.

Выбранный подход (ленивый upsert в middleware) решает это автоматически: любой
пользователь с валидным cookie при первом же запросе после деплоя проходит через
middleware и заводится в БД. Отдельных действий (принудительный релогин,
инвалидация cookie) не требуется.

## Поток данных

1. Запрос приходит в приложение, проходит аутентификацию → `context.User`
   заполнен из cookie.
2. `UserProvisioningMiddleware`: если аутентифицирован и username ещё не в
   кэш-гарде → `EnsureUserExistsAsync(username)`.
3. `UserRepository`: нет записи → вставка; есть → ничего. Username заносится в
   кэш-гард.
4. Управление передаётся дальше; запрос обрабатывается как обычно.

## Обработка ошибок

Provisioning не должен ломать пользовательский запрос. Ошибку обращения к БД в
middleware логируем и **пропускаем запрос дальше** — пользователь продолжает
работу, запись будет довнесена при следующем запросе. Исключения наружу не
пробрасываются. Username при неуспехе **не** заносится в кэш-гард, чтобы попытка
повторилась.

## Тестирование

Юнит-тесты `UserRepository.EnsureUserExistsAsync` (EF InMemory или SQLite
in-memory, как в существующих тестах инфраструктуры):
- создаёт запись, если пользователя нет;
- идемпотентен — повторный вызов не создаёт дубль;
- параллельные/повторные вставки не плодят записи (уникальный индекс).

Тест кэш-гарда: повторный вызов для уже обработанного username не обращается к
репозиторию/БД.

## Вне рамок (YAGNI)

- Перенос настроек (`UserTheme`, `UserProfile`) из LocalStorage в БД.
- Дополнительные поля пользователя (DisplayName, email, LastLoginAt).
- Принудительный релогин / инвалидация существующих cookie.
- Админский список/CRUD пользователей.
